# 无独显情况下的方案调整

## 1. 调整结论

当前本机只识别到 Intel 核显，没有可用于 CUDA 推理的 NVIDIA 独显。因此第一阶段不应继续以“本地 ComfyUI + SD1.5/SDXL/FLUX 推理”作为主路线。

新的 MVP 路线调整为：

```text
云端低价图像生成 API
+ 本地 prompt 模板与任务管理
+ 本地图片后处理
+ 本地素材库与预览导出
```

本地机器继续承担确定性工作：裁切、缩放、调色板量化、透明背景处理、文件管理、预览页面。AI 生成本身交给远程 API。

## 2. 为什么要调整

1. Intel 核显不支持 NVIDIA CUDA，无法稳定运行 ComfyUI 常见的 SD/FLUX 工作流。
2. CPU 推理即使能跑，单张图耗时也会非常长，不适合做素材生成工具的交互体验。
3. 无独显情况下，本地部署大模型的时间成本和稳定性风险高于 API 成本。
4. MVP 阶段目标是验证“能否生成可用像素素材”，不是验证“本地模型部署能力”。

## 3. 新方案架构

```text
前端/命令入口
  -> Prompt 模板增强
  -> 云端图像生成 API
  -> 下载原始生成图
  -> 本地后处理
  -> 本地素材库保存
  -> 预览 / 下载 / 重新生成
```

建议模块：

```text
Meowasen
  app/ 或 src/
    prompt_templates/      # 素材类型模板
    providers/             # 图像 API 适配层
    postprocess/           # Pillow/rembg 后处理
    storage/               # 本地素材与元数据
    ui/                    # 简单 Web 页面或命令入口
  assets/                  # 生成结果
  docs/plans/              # 方案文档
```

## 4. API 选型建议

### 4.1 MVP 首选：低价 FLUX 托管 API

推荐优先评估：

```text
Replicate / fal.ai / BFL API 上的 FLUX schnell 或 klein/pro 低价模型
```

理由：

1. 单张成本低，适合大量试错。
2. 不需要本地 GPU。
3. 生成速度和稳定性比 CPU 本地推理更适合交互。
4. FLUX 系列 prompt 理解能力较好，适合先做概念草稿。

建议策略：

```text
草稿生成：低价模型，一次生成 2 到 4 张
用户选择：保留一张
精修生成：必要时用更高质量模型
后处理：全部本地完成
```

### 4.2 备选：OpenAI 图像 API

OpenAI 图像 API 可作为高质量备选，但中高质量单张成本更高，不建议作为默认批量试错通道。

适用场景：

1. 低价模型生成质量不足时。
2. 需要更强 prompt 理解时。
3. 少量关键素材精修时。

### 4.3 暂不建议：本地 ComfyUI 主生成

无独显情况下，ComfyUI 可以保留为未来扩展方向，但不作为 MVP 默认依赖。

## 5. MVP 范围调整

### 5.1 保留

1. 素材类型选择：角色、道具、图标、地块参考图。
2. Prompt 模板增强。
3. 生成 1 到 4 张候选图。
4. 本地后处理：裁切、居中、最近邻缩放、调色板量化。
5. 本地保存生成记录和 PNG。
6. 预览、下载、重新生成。

### 5.2 调整

1. “本地 ComfyUI 调用”改为“图像 API Provider 调用”。
2. “模型路径/LoRA 配置”改为“Provider、模型名、API Key、质量档位配置”。
3. “显存验收”改为“API 调用失败、超时、余额不足、限流等异常处理验收”。

### 5.3 延后

1. 本地 ComfyUI 安装脚本。
2. 本地 SD1.5/SDXL 模型下载。
3. Pixel Art LoRA 管理。
4. ControlNet/IPAdapter/AnimateDiff。
5. 动画帧和多视图一致性。

## 6. 后处理仍然是关键

无独显方案下，后处理更重要，因为云端模型通常输出的是较大分辨率图片，不一定是真正像素素材。

MVP 后处理至少包含：

1. 裁切主体。
2. 输出固定画布：32x32、64x64、128x128。
3. 最近邻缩放。
4. 调色板量化，例如 16/32/64 色。
5. 去除背景或保留透明背景。
6. 保存原图和处理后图片，方便回溯。

## 7. 成本控制规则

1. 默认每次生成 2 张，不默认生成 4 张。
2. 设置每日/每月调用上限。
3. 保存 seed、prompt 和原图，避免重复生成。
4. 同一 prompt 支持本地后处理参数重跑，不重新调用 API。
5. 草稿模型和精修模型分层配置。

## 8. 验收标准调整

1. 未安装 NVIDIA 显卡和 ComfyUI 时，系统仍可完成一次素材生成。
2. 用户配置 API Key 后，可以生成至少 1 张候选图。
3. API 调用失败时，系统展示明确错误原因，不产生空白任务。
4. 生成结果可以完成本地后处理并输出 PNG。
5. 系统保存原始生成图、后处理图、prompt、provider、模型名和调用时间。
6. 本地无网络或 API Key 缺失时，系统应提示配置问题。
7. 后处理失败时，系统保留原始生成图。

## 9. 未来升级路径

当后续有 NVIDIA 独显或可用云 GPU 时，可以把生成 Provider 切换为本地 ComfyUI：

```text
Cloud Provider -> ComfyUI Provider
```

因此第一版代码应抽象 Provider 接口，不把某一家 API 写死在业务逻辑里。

建议 Provider 接口包含：

```text
generate(prompt, negative_prompt, width, height, count, seed, options) -> images
```

第一版实现一个云端 Provider；未来再实现 ComfyUI Provider。

## 10. 推荐下一步

1. 保持 PRD 的素材类型、后处理和素材库目标不变。
2. 把第一阶段生成后端改成云端低价 API。
3. 先实现 Provider 抽象和一个低价 API Provider。
4. 先做命令行闭环：prompt -> API 出图 -> 后处理 -> 保存 PNG。
5. 闭环跑通后再做本地 Web 页面。
