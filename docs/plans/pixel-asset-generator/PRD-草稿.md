# AI 像素素材生成工具 MVP 需求方案

## 1. 背景与目标

当前目标是做一个低成本、可本地运行的 AI 像素素材生成工具，用于快速生成游戏项目中的角色、道具、图标和基础地块素材。用户本地显卡约 8G 显存，希望优先控制成本，因此第一阶段不采用高频付费图像 API，也不做模型训练。

MVP 目标是先验证：用户输入描述后，系统能生成可预览、可下载、可放进游戏项目试用的像素风 PNG 素材。

## 2. 范围

### 2.1 范围内

1. 支持本地 ComfyUI 工作流生成图片。
2. 第一阶段模型优先采用 SD1.5 + Pixel Art LoRA。
3. 支持素材类型选择：角色、道具、图标、地块参考图。
4. 支持文本描述生成 1 到 4 张候选图。
5. 支持基础后处理：裁切、居中、透明背景、最近邻缩放、调色板量化。
6. 支持导出 PNG，并保存生成记录。
7. 支持用户对候选图进行保留、废弃、重新生成。

### 2.2 范围外

1. 不在 MVP 阶段实现完整动画帧生成。
2. 不承诺多姿势角色严格一致。
3. 不做 tileset 无缝拼接自动校验。
4. 不训练自有模型或 LoRA。
5. 不接入付费系统、多人协作、素材商城。
6. 不使用未经授权的第三方素材包训练或复刻风格。

## 3. 用户角色与场景

### 3.1 目标用户

1. 独立游戏开发者：需要快速生成可试用的像素素材。
2. 原型开发者：需要低成本生成占位素材，验证玩法和视觉方向。
3. 美术协作人员：需要批量探索风格方向，再进行人工精修。

### 3.2 使用场景

1. 用户想生成一个 64x64 像素风骑士角色，用于 RPG 原型。
2. 用户想生成一组宝箱、药水、钥匙等道具图标。
3. 用户想生成草地、石路、木地板等地块参考图。
4. 用户想在同一风格下多次生成候选素材，并保留满意结果。

## 4. 用户任务流

1. 用户进入生成页面。
2. 用户选择素材类型。
3. 用户输入素材描述，例如“蓝色披风的小骑士，俯视角，像素风”。
4. 系统根据素材类型补全 prompt 模板。
5. 系统调用本地 ComfyUI 工作流生成候选图。
6. 系统对生成图做后处理，输出透明 PNG。
7. 用户预览候选图。
8. 用户选择保存、下载、重新生成或废弃。
9. 系统保存生成记录，包括 prompt、模型、seed、素材类型、输出文件路径。

## 5. 功能需求

### 5.1 素材类型选择

系统应支持以下素材类型：

1. 角色：默认单主体、居中、透明背景、适合 64x64 或 128x128 输出。
2. 道具：默认单主体、无背景、轮廓清晰。
3. 图标：默认正方形构图，适合 UI 或背包图标。
4. 地块参考图：默认生成可作为 tile 设计参考的材质图，不承诺自动无缝。

### 5.2 Prompt 模板

系统应按素材类型自动拼接模板，至少包含：

1. pixel art 风格约束。
2. 清晰轮廓约束。
3. 简洁背景或透明背景倾向。
4. 固定视角约束。
5. 限制过度细节和写实渲染。

### 5.3 本地生成

系统应通过 ComfyUI 调用本地图像生成工作流。MVP 默认模型为 SD1.5 + Pixel Art LoRA。系统应允许配置模型路径、LoRA 名称、生成尺寸、步数、CFG、seed。

### 5.4 后处理

系统应对生成结果执行基础后处理：

1. 自动裁切主体外接区域。
2. 将主体放置到固定画布中心。
3. 支持输出 32x32、64x64、128x128 三种尺寸。
4. 使用最近邻缩放，避免模糊插值。
5. 支持调色板量化，降低颜色数量。
6. 支持透明背景；如果自动抠图失败，应保留原图并提示。

### 5.5 结果管理

系统应保存每次生成任务的结果，包括：

1. 原始 prompt。
2. 模板增强后的 prompt。
3. 模型和 LoRA 信息。
4. seed 和生成参数。
5. 原图路径。
6. 后处理结果路径。
7. 用户是否收藏。

### 5.6 导出

系统应支持单张 PNG 下载。MVP 阶段可以只导出 PNG，不要求导出 Aseprite、Unity package 或 Godot resource。

## 6. 模型与技术路线建议

### 6.1 第一阶段推荐

建议采用：

```text
ComfyUI + SD1.5 + Pixel Art LoRA + Pillow 后处理
```

理由：

1. 8G 显存运行压力较小。
2. 本地运行成本低，适合大量试错。
3. SD1.5 生态成熟，像素风 LoRA 和工作流较多。
4. 后续可平滑升级到 SDXL 或 FLUX。

### 6.2 第二阶段备选

可以引入：

```text
SDXL + Pixel Art LoRA
```

适用于提升大图质量、场景参考图和更复杂道具表现。

### 6.3 第三阶段备选

可以评估：

```text
FLUX.1-schnell 量化版或 API
```

适用于更强 prompt 理解和高质量概念草稿。但本地 8G 显存运行体验可能较慢，因此不建议作为 MVP 主力。

## 7. 验收标准

1. 用户可以在页面或命令入口输入描述并发起生成。
2. 系统可以调用本地 ComfyUI 完成至少 1 张图生成。
3. 生成结果可以完成后处理并输出 PNG 文件。
4. 对角色、道具、图标三类素材，系统均可生成候选结果。
5. 输出文件能够按任务目录保存，并能找到对应 prompt 和参数记录。
6. 当 ComfyUI 未启动、模型缺失或生成失败时，系统给出明确错误信息。
7. 后处理失败时，系统保留原始输出，不中断整个任务。
8. 在 8G 显存机器上，单次生成任务可完成，不因默认参数导致显存必然溢出。

## 8. 边界情况与规则

1. 用户输入为空时，不允许发起生成。
2. 用户输入过长时，系统应截断或提示。
3. 同一任务生成多张候选图时，每张图应有独立文件名。
4. 用户重新生成时，应保留历史结果，除非用户主动删除。
5. 系统不得默认使用未授权素材作为训练数据或参考图。
6. 系统不承诺 MVP 输出能直接达到商用最终美术质量。

## 9. 依赖、假设与待确认问题

### 9.1 依赖

1. 本地已安装 Python 环境。
2. 本地可运行 ComfyUI。
3. 本地可下载并加载 SD1.5 基础模型和 Pixel Art LoRA。
4. 本地具备 8G 显存 GPU。
5. 后处理依赖 Pillow；透明背景可选依赖 rembg 或同类分割能力。

### 9.2 假设

1. 第一版主要服务个人本地使用，不要求公网部署。
2. 第一版用户可以接受生成速度较慢。
3. 第一版以 PNG 静态素材为核心，不以动画为核心。
4. 第一版不需要账号、权限、计费和团队协作。

### 9.3 待确认问题

1. 第一版入口做成命令行工具、Web 页面，还是桌面应用？
2. 是否需要内置 ComfyUI 安装脚本？
3. 是否允许使用第三方开源 Pixel Art LoRA？
4. 默认输出尺寸优先 64x64 还是 128x128？
5. 是否需要项目级素材库目录结构，例如按角色、道具、地块分类保存？
6. 是否需要后续支持 Godot 或 Unity 的导出格式？
